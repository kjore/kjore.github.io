DBAIOps 中知识图谱的 schema

DBAIOps 构建的是一个 面向异常诊断经验的异构知识图谱 。其 schema 可以概括如下。

节点类型

  • Trigger Vertex ：异常触发节点。
    当系统识别到数据库中发生某类异常后，会先定位到对应的 Trigger 节点，并以该节点作为诊断入口，进一步搜索与之相关的其他节点，获取后续诊断所需的信息。
  • Experience ：经验节点。
    表示用于异常分析的经验知识，包括异常解释、可能原因、解决方案以及背景知识等。
  • Tool ：工具节点。
    表示面向特定异常诊断的可执行工具，通常是 Python 脚本或其他分析程序，用于自动化收集指标、验证假设、补充诊断证据。
  • Metric ：指标节点。
    表示诊断过程中关注的指标项或统计项，例如 CPU 使用率、内存使用率、查询响应时间、等待时间等。
    这里更偏向“指标概念 / 统计项”，而不是每一次实时采样值本身。
  • Tag ：标签节点。
    用于标识和归类指标、经验、工具等知识片段，并增强图的连接性。很多经验、指标、工具虽然名称不同，但可能属于同一类问题，Tag 节点负责将它们组织到同一语义空间中，形成更紧密的关联网络。
  • Auxiliary ：辅助节点。
    用于补充描述指标或诊断对象的附加信息，帮助系统更准确地理解某个指标，例如采样频率、统计方式、百分位数等。

边类型

  • Containment Edge ：表示包含关系。
    例如，一个 Trigger Vertex（如严重等待超时的 LOG_FILE_SYNC 异常）可以包含与之相关的 Experience Vertex（例如关于 redo log 分析步骤的经验说明）。
  • Relevance Edge ：表示相关关系。
    主要用于连接 Trigger Vertex 和 Metric Vertex，表示某个异常与哪些指标相关。
    例如，平均等待时间这一 Metric Vertex 与某个 Trigger Vertex 相关。
  • Diagnosis Edge ：表示诊断使用关系。
    表示某个 Experience Vertex 在诊断过程中会使用某个 Metric Vertex。
    例如，用于数据库 I/O 分析的 ash_db_io_ana 经验节点，会用到 db file sequential read wait time 这一指标节点。
  • Synonym Edge ：表示语义等价关系。
    主要用于连接两个语义相同但表达不同的 Tag Vertex。
    例如，physical_read 和 disk_read，shared_pool 和 shared_buffer。
    该类关系的作用是统一语义等价标签，避免图谱碎片化，增强相关经验和知识片段的整合能力。

DBAIOps 中知识图谱的构建方式

DBAIOps 的知识图谱并不是一次性从文档中抽取出所有实体和关系，而是围绕异常诊断过程逐步构建出来的。其整体流程可以概括为：

  1. 先人工定义一批 Trigger Vertex ，作为图的入口节点；
  2. 围绕这些 Trigger Vertex 自动抽取相关节点 ，包括 Experience、Tool、Metric、Tag 等，并同时建立基础关系，从而形成初始图结构；
  3. 进行图增强 ，主要利用 Tag 节点以及统计相似性等信息增强图的连接性，打通原本没有直接连接的节点；
  4. 根据新的异常报告和诊断经验持续更新图谱 ，使图谱能够随着新知识不断演化。

因此，DBAIOps 的知识图谱本质上是一个 面向异常诊断流程的动态图谱 ，而不仅仅是一个静态知识库。

总体来看，DBAIOps 中构建的知识图谱是一个 面向异常诊断经验的知识图谱 。
它的核心语料并不只是官方文档，而更依赖于：

  • 异常诊断经验总结
  • 历史案例分析报告
  • 故障处理手册
  • 可执行诊断脚本
  • 经验化的排障知识

也就是说，这个知识图谱不仅表示了“数据库中有哪些知识”，还把“ 诊断流程本身 ”建模进去了。因此，它更像一个 诊断决策图谱 ，而不是单纯的静态知识库。

我们当前希望构建的是一个 面向参数级别的知识图谱 ，重点在于表示：

  • 参数之间复杂的依赖关系
  • 参数与指标之间的影响关系
  • 参数约束、联调关系
  • 参数作用机理

从这个角度看，当前构建的图谱更接近一个 静态知识库 ，重点是“参数知识的表达与组织”；
而 DBAIOps 的图谱更接近一个 诊断知识图谱 ，重点是“异常如何被分析、证据如何被组织、诊断路径如何被扩展”。

我们的最终目标仍然是服务于 异常诊断问题 ，因此可以借鉴 DBAIOps 的思路，在现有参数知识图谱基础上逐步引入“诊断入口”和“诊断流程建模”。

可以参考 DBAIOps，在当前参数知识图谱中新增 Trigger Vertex ，将其作为异常诊断的入口节点。
之后再将 Trigger 与相关节点连接起来，形成一条从“异常现象”到“参数分析”的路径。

一个最基本的结构可以是：

Trigger → Metric

因为在当前图谱中，Metric 本身已经与 Parameter 建立了关系，所以顺着：

Trigger → Metric → Parameter

就可以从异常触发点逐步定位到与该异常可能相关的参数。

这意味着，现有图谱可以从“参数中心”扩展为“异常入口驱动的参数诊断图谱”。

Tool 节点如何引入

这个问题还有待解决

是否可以引入 Experience 节点

这一点我认为是可以借鉴，而且很有价值的。

目前我的想法是，把经验拆分为：

  • 参数节点
  • 公式节点
  • 指标节点

例如图中黄色节点表示一个公式关系，即：

参数 1 的推荐值由参数 2、3、4 的和决定

这种方式的优点是结构清晰，公式本身被显式表示出来。
但如果借鉴 DBAIOps 的思路，也可以考虑进一步将这类“公式节点”提升为 Experience 节点 。

也就是说，不再把它单独理解为纯公式，而是理解为一条完整的经验知识

这样的话，经验节点中可以同时包含：

  • 公式描述
  • 推荐值描述
  • 背景解释
  • 使用条件
  • 风险提示

然后再由 Experience 节点连接到相关的 Parameter 节点和 Metric 节点。

因此，一个可能的演化方向是：

  • 保留公式作为 Experience 节点中的一部分内容；
  • 将原本孤立的公式节点升级为更完整的经验节点；
  • 经验节点再连接参数、指标、触发条件和工具节点。
